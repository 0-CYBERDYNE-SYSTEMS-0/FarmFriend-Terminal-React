Autonomy Dashboard — E2E Loop vs. Ralph Wiggum Loop

Focus: long‑term autonomy, completion validation, and the trade‑off between deterministic testing and extreme long‑horizon execution.

Intent & Framing

This dashboard is conceptual. It does not change code and does not assume extra features beyond what is known about the current E2E loop and the Ralph Wiggum loop pattern.

E2E Loop (ff-terminal)

Deterministic Runtime Harness

Strengths for Autonomy

  • Uses real runtime + tool wiring
  • Explicit stop hooks and tool validation
  • Strong JSONL logging for post‑mortems

Long‑Horizon Limits

  • Session is per prompt loop, not perpetual
  • No built‑in multi‑day watchdog
  • Stops after a single prompt completes

Ralph Wiggum Loop

Perpetual Iteration Pattern

Strengths for Autonomy

  • Continuous loop until completion signal
  • File‑based “state of the world”
  • Heuristics for completion + rate limits

Long‑Horizon Limits

  • Less deterministic in testing contexts
  • Relies on completion promises and heuristics
  • Cost and drift risk without strict guardrails

Completion Validation — Side‑by‑Side

Dimension E2E Loop Ralph Loop
Primary stop logic Hook‑based stop check when no tool calls Completion promise / heuristic stop signal
Evidence of completion Hook evaluation + tool results in session logs Textual completion promise + task files
Risk of premature stop Lower (explicit hooks) Medium (heuristic‑driven)
Risk of infinite loop Lower (max iterations + stop hooks) Higher if completion signal never appears

Autonomy “Wealth Lag” View

Short‑Horizon Value

E2E loop returns fast validation. Ralph loop is heavier for short tasks.

Mid‑Horizon Value

E2E loop remains reliable; Ralph loop starts to pull ahead when tasks need multiple passes.

Long‑Horizon Value

Ralph loop dominates if the goal is days‑long persistence with minimal human input.

Spec Proposal: “Ralph‑Mode” as an Optional Operation Mode

Goal

Add a mode that keeps ff‑terminal running across long horizons by re‑invoking the agent until completion criteria are met, without replacing the E2E loop.

Core Idea

  • Persist a PROMPT.md and TASKS.md in the workspace.
  • Define a completion promise string that must appear to stop.
  • Loop the headless run with a watchdog and explicit rate limits.

Operational Safeguards

  • Hard daily token budget or run count budget
  • Time‑boxed iterations (e.g., 20–60 min)
  • Mandatory “checkpoint” file updates every N loops
  • Abort on repeated tool failures or unchanged task files

Ralph‑Mode Flow (spec)

Ralph‑Mode (ff-terminal concept)

PROMPT.md + TASKS.md
   │
   ▼
ff-terminal run --headless
   │
   ├─ runAgentTurn
   ├─ write/update TASKS.md
   └─ emit completion promise
   │
   ▼
Completion promise present?
   ├─ yes → STOP
   └─ no  → sleep/backoff → repeat

Recommendation Summary

  • Keep E2E loop as your core, deterministic runtime validation harness.
  • Consider a Ralph‑style mode for extreme long‑term autonomy if you can enforce budgets and completion criteria.
  • If added, treat Ralph‑mode as an optional “autonomy daemon,” not a replacement for E2E.
