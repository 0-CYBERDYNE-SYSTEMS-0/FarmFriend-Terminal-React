Toolchain Bridge: Quad-bop vs FF-Terminal

This dashboard maps your "code X" and "quad code" toolchain concept into FF-Terminal's tool registry + skill policies. It is structured as a bridge so you can decide whether to add a pre-session tool-gating layer or unify tools across workspaces.

Assumed Quad-bop Pattern

Toolchain Composition

  • Code X: low-level tool execution and OS affordances.
  • Quad code: higher-level orchestration and policy.
  • Pre-session handshake determines allowed tools.
FF-Terminal Tools

Registry + Skill Policy

  • Tool registry defines available tools.
  • Skill allowed-tools hook gates tool calls per session.
  • Skill sequencing can orchestrate multi-step flows.
Open Question

Where to Inject Tool Gating

  • Pre-session: choose toolset before any user turn.
  • In-session: load skill to update allowed tool list.
  • Hybrid: pre-session baseline + skill refinement.

Bridge Map (Conceptual)

Code X Runtime executors (run_command, edit_file, web tools)
Quad code Skill layer + allowed-tools policy + sequencing
Session Gate Pre-session tool manifest + workspace selection

Potential FF-Terminal Enhancements

1) Add a "tool manifest" file in workspace (TOOLS.md or tools.json) 2) Load manifest before the first turn to set allowed tools 3) Allow skills to extend or shrink manifest per task 4) Log tool policy changes in workspace logs

Path + Tool Contracts

  • Declare canonical asset roots independent of cwd.
  • Bind tool execution to workspace-defined roots.
  • Expose tool policy in UI (read-only or editable).
