Runtime Flow Comparison

ClawdBot Channel Inbound -> Gateway -> Runtime -> Channel Outbound WebChat/Control -> Gateway -> Runtime FF-Terminal CLI/Web UI -> Daemon -> Runtime -> Stream back WhatsApp -> Daemon (embedded) -> Runtime

Gateway abstraction yields isolation, easier restarts, and clear control-plane boundaries. The daemon model is lean but couples channel lifecycle to the runtime loop.

Operational Signals to Track

  • Channel up/down (WhatsApp, Telegram, Discord, WebChat)
  • Session handoff success rate (WS reconnect, WhatsApp pairing)
  • Tool denial events (skill allowed-tools hook)
  • Gateway queue depth vs. daemon streaming latency
  • Workspace health: missing logs, session files, allowlist drift

Delta Opportunities

  • Introduce a minimal gateway process to host channels and expose a clean WS API.
  • Move WhatsApp session cleanup and pairing review into a control UI.
  • Add a channel health heartbeat in the daemon and publish to workspace logs.
  • Implement group mention-only policy to reduce noise in shared chats.

Control Plane Checklist

  • Channel registry (connected, auth state, last error)
  • Session table (id, owner, last activity, tool policy)
  • Ops actions: restart channel, revoke pairing, rotate keys
  • Audit stream (tool blocked, pairing approved, session reset)

Staging-to-Prod Hardening

  • Isolate gateway process with system service (pm2/systemd)
  • Make workspace directory a first-class config artifact
  • Expose health endpoints for dashboards and alerting
  • Codify uptime and restart policy in scripts
