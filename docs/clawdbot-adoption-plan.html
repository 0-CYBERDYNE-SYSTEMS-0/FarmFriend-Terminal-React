ClawdBot -> FF-Terminal Adoption Plan

Planning-only document. Focus: agriculture-first UX, continuous sessions, and a ClawdBot-style workspace contract. This plan lists what to adopt, what to modify, and what to omit from FF-Terminal to reduce friction and improve trust.

Strategic Thesis

Why this adoption matters

  • A single long-lived workspace + session creates continuity and trust.
  • Explicit, human-readable workspace files make the system inspectable.
  • A gateway-first runtime stabilizes multi-channel messaging and ops.
Agriculture User Need

Low-tech, high-trust experience

  • Zero-config start (WhatsApp-first), immediate practical value.
  • Plain language and "magic wand" defaults; no tool jargon.
  • Daily check-ins, reminders, and safety confirmations.
Adoption Principle

One source of truth per memory layer

  • Use file contract for human-readable state.
  • Keep structured stores for machines, but derive summaries.
  • Avoid duplicate memory files without sync policy.

Session Model Shift

Current FF-Terminal
workspace -> per-session id -> artifacts per project
ClawdBot Model
single agent workspace -> long-lived "main" session
Target State
canonical workspace + stable session id across projects

Gateway Model Shift

Current
daemon + embedded WhatsApp
ClawdBot
single Gateway owns all channels + WS API
Target
Gateway abstraction for WhatsApp-first onboarding

Adopt / Modify / Omit

Area Adopt Modify Omit / Defer
Runtime Gateway-first channel ownership Session routing to "main" for direct chats Multi-channel expansion before gateway is stable
Workspace Bootstrap + contract files Daily memory logs + long-term memory split Parallel memory stores without sync
UX WhatsApp-first onboarding Plain language prompts + confirmations Advanced config in first-run
Safety Tool manifest guidance file Enforce allowed tools from manifest + skills Unbounded tool access in low-trust contexts

Phased Implementation (Planning Only)

Phase 0 Define canonical workspace + stable session id policy.
Phase 1 Add workspace contract files and bootstrap templates.
Phase 2 Introduce Gateway abstraction for WhatsApp-first onboarding.
Phase 3 Add daily memory logs and readable summaries.
Phase 4 Add tool manifest enforcement + skill metadata gates.
Phase 5 Migrate existing workspaces + UX polish for agriculture.

Risks & Mitigations

Risk Duplicate memory stores -> Mitigation single source of truth in MEMORY.md.
Risk Session fragmentation -> Mitigation canonical workspace + stable session id.
Risk Ops complexity -> Mitigation Gateway UI + health surface.

Acceptance Criteria

  • WhatsApp-first start works without local config.
  • Direct chats map to a single "main" session by default.
  • AGENTS/SOUL/TOOLS/USER/IDENTITY/MEMORY/PLAN/TASKS/LOG exist in workspace.
  • All memory and task views are readable and consistent.
