FF-Terminal UI Color + Hierarchy Guide (Ink/ANSI)

This is a “semantic” styleguide for the Ink CLI. Ink uses ANSI color names (the terminal’s theme defines the actual RGB), so we optimize for reliable contrast hierarchy rather than fixed hex values.

Reality check: across iTerm2/Terminal.app/Windows Terminal/themes, cyan may not match these swatches exactly, but “bright vs normal” typically preserves relative prominence. Keep the palette small (ANSI 16 colors) to stay stable.

Semantic Roles (what Ink uses)

Role Ink Color Token Notes
User input cyan Always user-authored; highest “non-error” emphasis.
Assistant response green Primary content output; readable and calm.
Thinking / scratch magenta Optional visibility; should not outshine assistant output.
Tool calls yellow “Activity” emphasis; short lines. Prefer symbols like ▶/■.
Errors red Reserved for failures only. Don’t use for warnings.
System / meta gray Low-priority status (“connected”, separators, hints).
Spinner cyanBright Single glyph with brackets (e.g., [|]) to stay visible.

Hierarchy Rules (portable across themes)

Use color to communicate meaning, and brightness to communicate priority. If a theme is “weird”, the user can still parse the UI by role.

  • bright* only for short, high-salience UI (spinner, banners), not for paragraphs.
  • Prefer gray (not dim) for meta so it stays readable on low-contrast themes.
  • Reserve red for real errors; avoid “red spam”.
  • Tool output should be mostly uncolored; color the tool boundary lines only.

Source mapping currently lives in ff-terminal-ts/src/cli/app.tsx (Transcript + Spinner).
