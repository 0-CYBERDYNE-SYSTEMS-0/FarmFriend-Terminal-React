Dual-Process Design

Separation of daemon and UI for stability and flexibility

Overview

FF Terminal uses a dual-process architecture where a long-running background daemon handles all heavy operations (tool execution, AI inference, state management) while the frontend processes (CLI, web) serve as lightweight clients that communicate via WebSocket.

Architecture Diagram

Component Overview

  • Daemon Process: Runs in background, handles all state and execution
  • CLI Process: Terminal interface, connects to daemon via WebSocket
  • Web Process: Browser interface, connects to daemon via WebSocket
  • WebSocket Server: Bidirectional communication layer

Process Responsibilities

Daemon Process

  • Manages agent state and context
  • Executes tools and skills
  • Handles AI provider communication
  • Maintains session persistence
  • Runs autonomy loops
  • Manages workspace isolation

CLI Process

  • Renders terminal UI
  • Handles user input
  • Displays agent output
  • Sends commands to daemon
  • Displays streaming responses

Web Process

  • Provides browser-based UI
  • Real-time chat interface
  • Interactive tool execution
  • Session history visualization
  • Settings and configuration

Benefits

  • Stability: Daemon crash doesn't affect UI, can reconnect automatically
  • Flexibility: Multiple UI clients can connect to same daemon
  • Performance: UI remains responsive during heavy computations
  • Scalability: Can run daemon on remote server, client locally
  • Testing: Easier to test daemon and UI independently

Communication Flow

typescript 
// Client → Daemon
{
  "type": "tool_execution",
  "tool": "read_file",
  "args": { "path": "./src/index.ts" }
}

// Daemon → Client (streaming)
{
  "type": "chunk",
  "content": "import { Agent } from './agent';"
}

// Daemon → Client (complete)
{
  "type": "complete",
  "result": { ... }
}

Startup Sequence

  1. Daemon starts on configured port (default: 28888)
  2. Daemon loads skills and initializes tool registry
  3. CLI/web client connects to daemon via WebSocket
  4. Session is established with unique UUID
  5. Client sends first command
  6. Daemon processes and streams response

Graceful Shutdown

The daemon supports graceful shutdown on SIGTERM/SIGINT:

  • Stop accepting new connections
  • Complete in-progress tool executions
  • Save session state to disk
  • Close all WebSocket connections
  • Exit with status code 0
