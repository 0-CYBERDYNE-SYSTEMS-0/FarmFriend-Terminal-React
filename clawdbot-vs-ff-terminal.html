1. Executive Summary

Clawdbot (Public Repo)

Always-on gateway for a personal AI assistant spanning many messaging channels (WhatsApp, Telegram, Slack, Discord, Signal, iMessage, Teams, WebChat), with control UI, canvas/visual surfaces, voice features, multi-agent routing, and a plugin/skills ecosystem.

Gateway control plane + WebSocket protocol

Skills registry + plugins + workspace skills

Sandboxing options for non-main sessions

FF-Terminal (This Repo)

AI terminal with daemonized runtime, Ink-based CLI UI, Web UI/FieldView, tool registry, scheduling/autonomy loops, and multi-provider model support.

CLI orchestrator + daemon

Tooling/skills + macOS automation KB

Scheduler + autonomy loop

Bottom line: both systems implement a daemon + tool-aware agent runtime, but Clawdbot is a broader multi-channel gateway product with companion apps, whereas FF-Terminal is a developer-focused terminal/web UI with strong local tooling and workflow automation.

2. Key Similarities

Daemon + Client UI

Both run a persistent server process and connect via clients (CLI/web) using WebSocket streaming.

Tools & Skills

Both expose structured tools/skills for actions, and use workspace-based instruction files.

Sessionized Agent Loop

Both maintain sessions for conversations and stream chunks as the model responds.

Extensible UI Layer

Both provide web UI surfaces and expansion paths (FieldView vs Control UI/Canvas).

Security & Policies

Both have explicit policy layers (tool gating, sandboxing/allowlists, session controls).

Automation & Scheduling

Both include background runs: FF-Terminal scheduler/autonomy; Clawdbot heartbeat/cron.

3. Key Differences

Scope & Surfaces

Clawdbot targets a multi-channel messaging gateway (WhatsApp/Telegram/Slack/Discord/Signal/iMessage/Teams/WebChat), whereas FF-Terminal targets a local dev/operator experience (CLI + web + FieldView + ACP).

Companion Apps & Nodes

Clawdbot ships optional macOS/iOS/Android apps and “nodes” with camera/screen/voice surfaces. FF-Terminal focuses on local tools (including macOS automation scripts) rather than mobile app nodes.

Gateway vs Daemon

Clawdbot’s gateway exposes presence, config schema, webhooks, and a control UI; FF-Terminal’s daemon is a lean WS runtime for agent turns and tool streaming.

Plugin Model

Clawdbot supports runtime plugins/extensions (TS modules loaded via jiti). FF-Terminal focuses on built-in tools and skills registered at runtime.

Security Defaults

Clawdbot emphasizes DM pairing and optional Docker sandboxing for non-main sessions. FF-Terminal emphasizes local tool guards and explicit allow flags (e.g., macOS control).

Configuration Style

Clawdbot uses a JSON5 config at ~/.clawdbot/clawdbot.json with schema exposure. FF-Terminal uses config JSON plus profile/credential storage and env overrides in the workspace.

5. Architecture Focus

Clawdbot Emphasis

  • Multi-channel routing and safety policies
  • Gateway protocol + schema-driven clients
  • Companion apps, nodes, canvas, voice
  • Plugin system and external integrations

FF-Terminal Emphasis

  • Developer-centric workflow & tooling
  • Ink terminal UI + web/FieldView
  • Tool registry with local filesystem automation
  • Autonomy loop + scheduled tasks

6. Tradeoffs & Implications

  • Scope vs Focus: Clawdbot’s breadth across channels/apps increases operational complexity; FF-Terminal’s focus enables simpler local automation.
  • Surface Area: Clawdbot’s gateway exposes more integration points (webhooks, control UI, nodes), while FF-Terminal optimizes a tighter tool surface.
  • Security Posture: Clawdbot’s multi-channel entry points require stricter default policies; FF-Terminal is optimized for single-user local control.
  • Extensibility: Clawdbot’s plugin system enables deep gateway customization; FF-Terminal’s tool registry favors rapid local extension.
